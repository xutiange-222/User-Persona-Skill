# 公共步骤:研究目标对齐

这一步在 SKILL.md 的 Step 0/1 中被引用,是所有范式启动后第一件必做的事。

---

## 为什么必做

研究目标是后续所有"什么重要 / 怎么聚类 / 怎么选 / 突出什么"判断的**唯一合法锚点**。

没有锚就开始,模型只能从访谈数据里"自己找重点",结果一定是数据驱动的归纳,不是用户驱动的价值。这违反 SKILL.md 硬约束 1。

---

## 必须问清的三件事

| 问题 | 用途 |
|------|------|
| 1. 这份画像主要给谁看?(读者) | 决定画像内容的抽象层次、术语难度、详略 |
| 2. 它要回答什么问题?(研究问题) | 决定哪些字段必要、哪些字段可以省 |
| 3. 它要支持什么决策?(用途) | 决定画像内容的方向(产品功能定位 / 市场策略 / 内部团队对齐 / ...) |

---

## 问的话术
## 工作目录命名提醒

开场时必须说明两个文件夹的区别:

- `过程稿/`:保存中间过程,用于继续跑和排查问题
- `最终交付件-<对象类型>-<项目名>-<样本数>用户-<构建方式>/`:保存最终 HTML 和依赖,用于复制、转发、归档

项目名默认从研究目标、输入文件夹名或报告标题中提取;太长时压缩到 6 到 18 个字。构建方式用用户能理解的短词,如 `单画像`、`多画像`、`2维`、`多区分点`、`多角色旅程`、`双页画像`。

```
我会在当前操作目录下创建 `用户画像报告输出/<项目名>-<日期时间>/`。
里面包含 `过程稿/`、`画像头像素材/`、`界面截图/` 和 `最终交付件-<对象类型>-<项目名>-<样本数>用户-<构建方式>/`。字段对齐时我会主动问你头像和典型场景要不要配图(见 `steps/visual-assets.md`)。`过程稿/` 放研究目标、构建方式、字段选择、抽取结果和合并结果,用于断点续跑。`最终交付件-*` 放可以整包复制的 HTML 和依赖,例如 `最终交付件-2C-华为云音乐HiRes专区-5用户-2维`。

在开始之前,我需要先和你确认这份画像报告的研究目标。这三件事会决定后面所有的侧重点:

1. **这份画像主要给谁看?**
   例:产品团队 / 设计团队 / 运营团队 / 公司高层 / 跨部门同步

2. **它要回答什么问题?**
   例:"哪类用户对功能 X 最敏感"
        "运维团队每天最关心什么"
        "新功能应该优先满足哪类人的需求"

3. **它要支持什么决策?**
   例:"决定新功能要不要做"
        "决定要不要拆分服务团队"
        "决定营销重点投在哪个细分人群"

你可以一次性回答,也可以分次回答。如果还没想清楚,我们可以先讨论一下。
```

**顺序约束**:
- 研究目标三问必须先完整问完,再介绍画像构建方式。
- A/B/C/D/E 首次出现时,必须逐条解释"适合什么情况 + 会产出什么",不能只写字母让用户选。
- 只有在本轮消息里已经问过的问题,才能要求用户回复。不能在没有展示方式说明时,要求用户回复"A/C/D/E"。

---

## 用户回答模糊时怎么办

用户经常会给出模糊回答,例如:

- "给团队看" → 哪个团队?具体哪几个人?
- "了解用户" → 了解什么?决定什么?
- "支持产品决策" → 什么决策?功能层?定位层?

**模型必须追问**,不要把模糊回答硬当成清晰锚点。追问时给具体选项,不要开放性提问。

例:

```
"给团队看" — 是产品经理团队、研发团队、还是混合团队?
            他们看完这份画像,主要想用它做什么?
            (示例:决定下一版功能优先级 / 设计新流程 / 培训销售...)
```

---

## 用户给不出清晰锚的兜底

如果用户反复给不出清晰研究目标(比如说"我先看看再说""你帮我决定"),**模型不能继续往下走**。这违反硬约束 1。

兜底话术:

```
我理解你想先看看效果,但这份画像后续每一步的判断都需要基于你的研究目标来做(哪些字段重要、怎么排版、突出什么)。如果没有这个锚点,我做出来的画像可能不符合你实际想用的场景。

建议两个做法:
1. 先选一个"假设的研究目标"开始,我们走一遍流程,你看到效果后再调整
2. 你回去和团队聊一下,确认了再来

你倾向哪个?
```

注意:这段兜底只能在已经完整问过研究目标三问之后使用。如果还没有问过,先补问三问。

---

## 锚点确认后的强制行为

研究目标一旦对齐:

1. **写入** `00-research-goal.json`,后续所有步骤启动时都要读一遍
2. **每个关键决策**(字段选择、区分点推荐、画像内容侧重)的模型话术里,**必须显式回扣**这个锚:
   - "基于你说的『给产品团队看,回答新功能定位问题』,我建议在画像里侧重职责和痛点字段"
   - "你说『要决定服务团队要不要拆分』,那我推荐的分类依据是『工作内容差异度』"
3. 锚后续可以更新,但任何更新都要重写 `00-research-goal.json`,并提示用户"研究目标变了,前面的字段对齐可能需要重新做"

---

## JSON 产物格式

```json
{
  "version": "v8",
  "created_at": "ISO 时间",
  "research_goal": {
    "audience": "产品团队",
    "audience_detail": "电站调度产品的 PM 和设计师,共 6 人",
    "research_question": "运维团队每天的工作压力点在哪里,以及对 AI 辅助决策的接受度",
    "decision_use": "决定下一版 AI 辅助决策产品的核心功能优先级"
  },
  "research_type": "toB",
  "notes": "用户原话补充信息,例:这是给季度评审会用的"
}
```

`research_type` 由模型基于访谈内容判断,但用户可以推翻。

`audience` 是简短描述,`audience_detail` 是用户提供的补充。`research_question` 和 `decision_use` 直接来自用户原话,模型只做轻微改写让其完整。

---

## 边界情况

- **多个读者群体**:`audience` 写主要群体,`audience_detail` 写其他次要群体。"主要给产品团队看,设计团队和运营也会看"
- **多个研究问题**:`research_question` 写最核心的一个,其他写在 notes。
- **决策还没定**:`decision_use` 写"探索性,尚未确定具体决策方向",这种情况下后续推荐字段时优先全面而不是侧重。
