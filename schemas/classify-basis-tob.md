# toB/toD 常见画像分类依据库

R3 场景下(toB/toD 文件结构不清晰,需要和用户对齐分类依据),这份文档给出**常见的分类依据**作为推荐选项。

模型推荐时**必须基于研究目标**回扣,不是把全部选项给用户挑。

---

## 常见分类依据(toB/toD)

### 1. 按岗位/角色分类

最常见。适用于:研究目标涉及多角色协同、决策链路、不同角色对产品的差异化需求。

例:
- 产品经理 / 设计师 / 研发 / 测试
- 调度员 / 运维 / 管理者
- 销售 / 客户经理 / 售前 / 售后

### 2. 按技能水平/经验分类

适用于:研究目标涉及产品功能分层、新手 vs 熟手的差异化体验、培训设计。

例:
- 初级(< 2 年)/ 中级(2-5 年)/ 高级(> 5 年)
- 新手 / 熟练 / 专家

### 3. 按组织层级分类

适用于:研究目标涉及决策权、向上汇报、跨层级沟通。

例:
- 一线员工 / 一线管理者(组长)/ 中层管理(经理)/ 高管(总监+)

### 4. 按业务领域/赛道分类

适用于:研究目标涉及行业 know-how 差异、产品要支持多赛道。

例:
- 金融 / 医疗 / 教育 / 制造
- C 端电商 / B 端 SaaS / 内部工具

### 5. 按企业规模分类(toB 销售场景)

适用于:研究目标涉及销售策略、定价、付费意愿。

例:
- 初创(< 50 人)/ 中小(50-500)/ 中大型(500-5000)/ 大企业(> 5000)

### 6. 按使用场景分类

适用于:研究目标涉及场景差异化设计。

例:
- 日常作业场景 / 应急处理场景 / 培训演练场景
- 远程办公 / 现场办公

### 7. 按团队/部门分类

适用于:研究目标涉及不同部门对同一产品的使用差异。

例:
- 销售部 / 市场部 / 客服部 / HR

---

## 推荐话术模板

R3 场景下,模型必须按这个结构推荐:

```
我看了你的访谈,这批受访者似乎角色比较多样。
基于你的研究目标(给 [audience] 看,回答 [research_question]),
我建议按 [推荐的分类依据] 来分。

理由:
- 你要回答的问题涉及 [回扣研究目标]
- 这种分类下,每个画像的差异点最能服务这个目标

具体的分类依据是:[列出具体类别]
- 类别 1:[名称],对应受访者:[名字]
- 类别 2:[名称],对应受访者:[名字]
...

这个分类你认可吗?
- 认可,直接用
- 调整某些受访者的归类
- 换一种分类依据(我可以列出其他选项给你看)
- 拆/合并某些类别
```

**不允许**模型让用户从"一堆并列选项里挑一个",必须先给出**带理由的推荐**。

---

## 分类依据下面可以有"价值变量"

一个分类依据可以包含 1~N 个**价值变量**(细化维度):

例:分类依据 = "按岗位/角色",价值变量可能有:
- 主要职责
- 决策权
- 与产品的交互深度

价值变量是分类依据的子层概念,**不会和"字段"混淆**(字段是画像里展示的信息块,价值变量是分类时用的判断维度)。

在 R3 场景下,价值变量主要用于**辅助模型判断每位受访者属于哪个类别**,通常不直接展示给用户。R4/R5 场景下,价值变量会显式展示(因为是 toC 数据驱动分群的核心)。

---

## 边界情况

- **用户原本是 R3 但想换成 R4/R5(数据驱动)**:R3 是基于已知分类依据分组,R4/R5 是基于数据归纳出分类依据。如果用户说"我也不知道按什么分,你帮我从数据看",模型应该**先确认是 toB 还是 toC**;toC **默认推荐** R4/R5;toB **默认推荐** R3(给出几个常见分类依据让用户选),但**用户确认后 toB 同样可用 D/E**,子页必须走 2B 组件族(`layout-2b-grid` + `tob_journey_l2`,总览页仅配色变化)
- **分类依据组合**:用户可以选"岗位 × 经验水平"这种组合,产生更细的分类(产品经理-新手 / 产品经理-熟手 / 设计师-新手 / 设计师-熟手)。组合分类适用于样本量大、需要精细颗粒度的研究
- **某个类别只有 1 个受访者**:照样保留,但模型要提示"该类别样本只有 1 人,内容可能单薄,要不要合并到相近类别"
